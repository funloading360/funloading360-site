🚨 Top 5 Critical Findings

Issues confirmed by multiple specialists with direct revenue or security impact

💰 Price Manipulation — Anyone Can Pay £0.30 for a £500 Service

totalPrice is sent from the client browser and used directly in Stripe checkout without server-side validation against the product catalog. An attacker sends {totalPrice: 2, paymentType: 'deposit'} and pays 30p instead of ~£150.

Confirmed by: SEC-01 · API-02 · CRO-07

🔄 Deposit Inconsistency: 15% vs 30% on the Same Page

PricingContent.tsx:319 shows "30% deposit required" but constants.ts defines DEPOSIT_PERCENT = 0.15, the FAQ says 15%, and booking flow calculates 15%. This trust-destroying discrepancy appears at the critical decision moment.

Confirmed by: UX-01 · CRO-01 · API-05

🔁 Complete Funnel Break: Pricing → Book → Redirect Back to Services

"Book This Package" on every pricing card links to /book. BookingFlow auto-redirects to /services if cart is empty. Result: every user from the pricing page is looped back to services without being able to book.

Confirmed by: UX-03 · MOB-02 · CRO-02

📧 Webhook Race Condition — Duplicate Confirmation Emails

Stripe webhook and /api/checkout/verify-session can both run simultaneously (0–30 second window). Both check booking.status non-atomically, both see 'pending', both send confirmation emails. Customer receives 2 emails, admin gets 2 notifications, n8n processes payment twice.

Confirmed by: SEC-03 · API-01

🎨 Design System is Dead — tokens.ts Imported by Zero Components

tokens.ts defines a complete design system (colors, spacing, typography, shadows) but is not imported anywhere. All components hardcode hex values. Brand color change requires updating 4+ files manually instead of 1.

Confirmed by: ARCH-03 · BRAND-01
